BUG INVESTIGATION:

"The bug report said inventory values on the dashboard become incorrect
after certain product management operations — the values are fine
initially but drift downward after normal use.

My first hypothesis was that the dashboard calculation was wrong, so I
checked index.js — the value formula looked correct. Then I asked: when
does the value change? After editing a product. So I went straight to
the PUT handler in products/[id].js."

  >> Show the original PUT block (lines 18–34)
  "Right here — this line was the culprit:
    unitCost: parseInt(req.body.unitCost)

  parseInt truncates to the integer part. So a product with a unit cost
  of 2.50 becomes 2 on the first edit, and 0.85 becomes 0. The corruption
  is silent — no error, no warning. Every time a manager edits any field
  on a product, the unit cost gets silently floored to its integer part.

  The fix is a one-word change: parseFloat instead of parseInt for
  unitCost. I left parseInt for reorderPoint, which is always a whole number."

  >> Show the fixed line
  "unitCost: parseFloat(req.body.unitCost)

  I also added a comment explaining what was wrong and why, so the next
  developer doesn't accidentally revert it."

SCALING WRITE-UP:

  >> Open README.md, scroll to '## Task 4B — Scaling Write-up'

  "I also wrote up how I'd evolve this architecture for 500 warehouses,
  10,000 products, and 50 concurrent users.

  The first thing that breaks is the JSON file storage. Every request
  reads the entire file, mutates it, and writes it back — with no
  concurrency control. Two simultaneous transfers would race each other
  and one would silently overwrite the other's changes.

  The fix: replace JSON files with PostgreSQL. Stock transfers become
  proper BEGIN/UPDATE/UPDATE/COMMIT transactions with row-level locking —
  eliminating the race condition entirely. Indexes on productId and
  warehouseId turn full-file scans into O(log n) lookups. Add Redis for
  caching the product catalog, PgBouncer for connection pooling under
  concurrent load, and Docker for horizontal scaling as the operation grows."

WHAT I'M PROUD OF:

"The piece I'm most satisfied with is the atomicity solution in the
transfer API. Using a single writeFileSync after applying both stock
changes in memory is the correct answer for this storage layer — it
doesn't require a database or a queue, it just uses the guarantee that
the OS won't give you a partial write on a single call. It's simple,
it's correct, and it's easy to reason about.

I'm also happy with how the three tasks connect: the transfer history
feeds directly into the velocity calculation, which feeds into the reorder
recommendations, which surface on the dashboard. It feels like a real
system rather than three disconnected features."

KNOWN LIMITATIONS:

"The main limitation is that the single-file atomicity trick only protects
within one request. Two simultaneous transfers can still race each other —
there's no file-level locking across requests. That's the core reason
I'd reach for PostgreSQL at scale.

The velocity calculation uses a 30-day window and treats all transfers
equally. A smarter version would weight recent transfers more heavily
and distinguish inbound from outbound movements.

Lead time is configurable per session but not persisted — refreshing
the page resets it to 7 days. In production this would be a per-product
or per-supplier setting stored in the database."

WHAT I'D IMPROVE WITH MORE TIME:

"I'd add optimistic locking on the stock writes — a version field that
gets checked before committing, so concurrent requests fail fast rather
than silently overwriting. I'd also add unit tests for the reorder
formula and the transfer validation, and TypeScript across the board
for type safety on the JSON shapes."
